## Research Question
Do MsExperiment objects with annotated sampleData$type produce identical screening-mode EIC diagnostic outputs compared to file-path-based invocation of tardisPeaks()?

## Connected Finding
The tardisPeaks() function accepts both file paths and MsExperiment objects as input when screening_mode=TRUE, with the MsExperiment approach requiring sampleData$type to be populated to distinguish QC from sample runs.

## Task Description
Construct an MsExperiment object from vignette mzML files with sample type labels, execute tardisPeaks() in screening mode using the MsExperiment object as lcmsData input, and verify that the resulting diagnostic EIC PNG files match those from file-path-based invocation.

## Inputs
- mzML files (centroided) corresponding to LC-MS runs for targeted analysis
- Target list data frame with columns: compound ID, compound name, m/z, expected retention time (minutes), and polarity
- Sample metadata including type labels (e.g., QC, sample) for each mzML file

## Expected Outputs
- EIC PNG diagnostic figures saved to output folder for visual inspection of target peak detection in screening mode
- Verification report confirming EIC PNG files from MsExperiment-based invocation match file-path-based reference invocation

## Domain Knowledge
- MsExperiment is a container class that combines Spectra objects with sample-level metadata, enabling integrated downstream analysis without separate file-path indexing.
- Screening mode in tardisPeaks() checks whether targets are detectable within their defined m/z and retention-time windows before performing full peak integration across all samples.
- EIC (Extracted Ion Chromatogram) PNG files are diagnostic outputs that visualize the actual chromatographic signal for each target, enabling manual quality assessment and troubleshooting of detection parameters.
- Centroiding converts raw profile-mode mass spectrometry data to centroid-mode by reducing each peak cluster to a single m/z–intensity pair, which is required for TARDIS input files in mzML format.
- API equivalence testing verifies that two functionally identical code paths (object-based vs. file-path-based invocation) produce identical or visually indistinguishable results, confirming correct wrapper implementation.
