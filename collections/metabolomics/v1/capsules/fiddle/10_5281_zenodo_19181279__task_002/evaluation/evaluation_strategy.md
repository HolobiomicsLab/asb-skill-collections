# Evaluation Strategy

## Direct Checks

- file_exists: verify that prepare_augment_rescore.py exists in the repository github:JosieHong__FIDDLE
- script_runs: execute prepare_augment_rescore.py with TCN train and test set inputs (concrete file paths from repository) and verify script completes without runtime errors
- file_format_is: verify output files are in expected format (CSV or pickle, as indicated by script logic)
- field_present: verify output training data contains columns/fields for: positive/negative labels, precursor m/z, formula, spectrum identifiers
- value_in_range: verify positive:negative ratio in output training data is 1:1 (robust to rounding; allow 0.95–1.05 ratio)
- value_in_range: verify capped positives per formula do not exceed --pos_cap parameter value specified in script invocation
- contains_substring: verify cross-spectrum negatives are generated by checking that negative pairs have distinct spectrum identifiers but fall within --tolerance ppm of precursor m/z
- row_count_equals: verify test split row count matches input test set (no augmentation applied to test); robust to any permutation of rows
- output_matches_reference: if a reference augmented dataset is deposited (Zenodo or repository release), compare row counts, ratio statistics, and negative-pair m/z distances to reference output (parameter-sensitive to random seeds and pool worker ordering; no canonical answer without fixed seed specification)

## Expert Review

- Assess whether the --pos_cap, --tolerance, and --num_workers parameter settings used are scientifically justified and appropriate for formula prediction rescoring
- Evaluate whether the 1:1 positive:negative downsampling ratio is optimal for training a BCE-loss Siamese rescore head, or if alternative ratios should be tested
- Assess whether the cross-spectrum negative generation strategy (m/z window within --tolerance ppm) introduces realistic or biologically plausible false-match scenarios
- Review whether formula refinement parallelization via multiprocessing.Pool preserves the correctness of the refinement logic and does not introduce non-deterministic artifacts
